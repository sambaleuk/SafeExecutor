## Phase 2 — Policy Engine + Default PostgreSQL Rules

**Goal:** Evaluate operations against policy rules. Implement the default policy that covers the most dangerous PostgreSQL patterns.

### Deliverables
- `src/core/policy-engine.ts` — Rule evaluation engine
- `schemas/policy.schema.json` — JSON Schema for policy files
- `config/default-policy.json` — Default rules for PostgreSQL

### Default policy rules (in priority order)
| Rule | Match | Action | Risk |
|------|-------|--------|------|
| `deny-delete-no-where` | DELETE + no WHERE | DENY | CRITICAL |
| `deny-truncate` | TRUNCATE | require_approval | CRITICAL |
| `deny-drop` | DROP | require_approval | CRITICAL |
| `alter-large-table` | ALTER | require_approval | HIGH |
| `delete-with-where-dry-run` | DELETE + WHERE | require_dry_run | HIGH |
| `update-no-where` | UPDATE + no WHERE | require_approval | HIGH |
| `update-with-where` | UPDATE + WHERE | require_dry_run | MEDIUM |
| `insert-allow` | INSERT | require_dry_run | LOW |
| `select-allow` | SELECT | allow | LOW |

### **[GATE]** Non-bypassable policy invariants
- DENY rules cannot be overridden at runtime (no `--force` flag)
- CRITICAL risk level always escalates to require_dry_run + require_approval
- Unknown operations are denied when `defaults.allowUnknown: false`

### Acceptance criteria
- [x] `DELETE FROM users` → PolicyDecision `{ allowed: false, riskLevel: 'CRITICAL' }`
- [x] `DELETE FROM users WHERE id = 1` → `{ requiresDryRun: true, riskLevel: 'HIGH' }`
- [x] `SELECT * FROM orders` → `{ allowed: true, riskLevel: 'LOW' }`
- [x] AJV validation rejects malformed policy files

## Phase 3 — Sandbox / Dry-Run Layer

**Goal:** Simulate operations before any real execution. PostgreSQL EXPLAIN ANALYZE + transaction rollback.

### Deliverables
- `src/core/sandbox.ts` — Dry-run orchestrator
- `src/adapters/postgres.ts` — PostgresAdapter with dry-run support

### Simulation strategies by operation type

```
SELECT   → EXPLAIN ANALYZE (read-only, no side effects)
DML      → BEGIN → EXECUTE → capture rowCount → ROLLBACK
DDL      → EXPLAIN only (DDL cannot be reliably rolled back in PG)
TRUNCATE → Schema inspection only (size estimation via pg_stat_user_tables)
```

### Warnings generated by sandbox
- Operation has no WHERE and would affect N rows
- Large operation (>10k rows)
- Estimated execution time exceeds threshold (from EXPLAIN ANALYZE)
- Referenced table does not exist

### **[GATE]** Sandbox invariants
- Dry-run transactions ALWAYS rollback — no path to commit
- DDL operations get a warning that simulation is incomplete
- If sandbox returns `feasible: false`, pipeline aborts — no execution

### Acceptance criteria
- [x] DML dry-run returns actual `rowsAffected` from rolled-back transaction
- [x] SELECT dry-run returns `estimatedRowsAffected` from EXPLAIN
- [x] No data is modified after a dry-run
- [x] `SandboxResult.warnings` populated for no-WHERE DML

## Phase 4 — Approval Gate

**Goal:** Route operations to the correct approval flow based on risk and config mode.

### Deliverables
- `src/core/approval-gate.ts` — Three approval modes: auto, cli, webhook

### Approval modes

**`auto` mode**
- LOW/MEDIUM → auto-approve
- HIGH → reject (must use cli or webhook)
- CRITICAL → always reject

**`cli` mode**
- Interactive terminal prompt
- Displays: risk level, operation type, tables, WHERE presence, row estimate, warnings
- Captures approver identity and timestamp

**`webhook` mode**
- POSTs `ApprovalRequest` to configured URL
- Waits for `ApprovalResponse` (with configurable timeout)
- Designed for Slack bots, PagerDuty, or custom approval apps

### **[GATE]** Approval invariants
- Rejected operations never reach the executor
- Approval records are immutable once captured
- Webhook timeout defaults to 300s — operations never hang indefinitely

### Acceptance criteria
- [x] `auto` mode approves LOW/MEDIUM, rejects HIGH/CRITICAL
- [x] `cli` mode prompts correctly and captures approver name
- [x] Webhook mode POSTs correct payload and handles timeout
- [x] Rejected approval → pipeline aborts with audit entry

## Phase 5 — Execution + Automatic Rollback

**Goal:** Safe execution wrapped in a transaction with savepoint. Auto-rollback if rows affected exceed prediction.

### Deliverables
- `src/core/executor.ts` — Savepoint-wrapped executor

### Execution flow

```
BEGIN TRANSACTION
  SAVEPOINT se_sp_<timestamp>
    EXECUTE sql
    check: rowsAffected <= estimatedRows * 1.5
    if exceeded → ROLLBACK TO SAVEPOINT → ROLLBACK TRANSACTION → abort
  COMMIT TRANSACTION
```

### Rollback triggers
1. Rows affected > `estimatedRows * 1.5` (configurable multiplier)
2. Executor throws an exception
3. `database.maxRowsThreshold` exceeded (from config)
4. Post-execution assertion fails (Phase 7 extension point)

### **[GATE]** Executor invariants
- Direct `execute()` call is not exposed in the public API
- Savepoints are ALWAYS used — no bare transaction execution
- Rollback reason is always logged in AuditEntry

### Acceptance criteria
- [x] Successful execution → `status: 'success'`, rows committed
- [x] Threshold exceeded → `status: 'rolled_back'`, rollbackReason set
- [x] Exception during execution → `status: 'failed'`, transaction rolled back
- [x] `savepointUsed: true` always in ExecutionResult

## Phase 6 — Audit Trail

**Goal:** Complete, tamper-evident log of every pipeline run — success or failure.

### Deliverables
- `src/core/audit.ts` — Audit writer (console, file, database)
- `schemas/audit.schema.json` — JSON Schema for audit entries

### Every AuditEntry captures
| Field | Description |
|-------|-------------|
| `id` | Unique entry ID |
| `timestamp` | ISO 8601 |
| `executor` | Service/user identity |
| `operation` | Full ParsedIntent |
| `policyDecision` | Which rules matched, risk level |
| `sandboxResult` | Row estimates, execution plan, warnings |
| `approvalResponse` | Who approved, when |
| `executionResult` | Status, rows affected, rollback reason |
| `totalDurationMs` | End-to-end pipeline duration |
| `environment` | From config |

### Output destinations
- **console** — human-readable summary + JSON in verbose mode (`SAFE_EXECUTOR_VERBOSE=true`)
- **file** — newline-delimited JSON, appendable log
- **database** — writes to `audit_log` table (Phase 8)

### Acceptance criteria
- [x] Every pipeline run (success + failure + abort) produces an AuditEntry
- [x] AuditEntry is valid against `audit.schema.json`
- [x] File output is appendable and parseable line by line
- [x] Verbose mode outputs full JSON

## Phase 7 — Adapter System

**Goal:** Clean interface that allows plugging in MySQL, SQLite, CockroachDB, or any custom database without modifying the core.

### Deliverables
- `src/adapters/adapter.interface.ts` — DatabaseAdapter interface (already defined in Phase 1)
- Upgrade `src/core/intent-parser.ts` to use `node-sql-parser` for better multi-dialect support
- Document adapter contract in README

### Adapter responsibilities
- Connection lifecycle (connect, ping, close)
- EXPLAIN / EXPLAIN ANALYZE
- Transaction management (begin, savepoint, rollback, commit)
- Dry-run transaction (execute + always rollback)
- execute() for the real operation

### Example: adding a MySQL adapter
```typescript
import type { DatabaseAdapter } from 'safe-executor';
import mysql from 'mysql2/promise';

export class MySQLAdapter implements DatabaseAdapter {
  readonly name = 'mysql';
  // ... implement all interface methods
}

const executor = new SafeExecutor({
  configPath: './config.json',
  adapter: new MySQLAdapter('mysql://...'),
});
```

## Architecture Decisions Log

| Decision | Rationale |
|----------|-----------|
| No `--force` or skip flags | Gate bypassing defeats the purpose of the harness |
| Savepoints on every execution | Allows rollback without aborting the outer transaction |
| AJV validation on config load | Fail fast at startup, not at execution time |
| Separate policy file from config | Policy changes need review; config changes are operational |
| regex parser in Phase 1 | Simple, zero deps. Upgrade to AST when multi-dialect needed |
| Approval modes: auto/cli/webhook | Covers dev, ops, and automated pipelines without coupling |
